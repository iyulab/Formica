# Formology 개요

**문서중심 개발 방법론** (The Science of Forms and Flows)

## 한 문장 정의

> Formology는 추상적 개념(Entity, Domain Model)이 아닌 **현업이 실제로 사용하는 구체적인 양식과 문서**(의뢰서, 보고서, 기록지)부터 시작하여, 자연스럽게 데이터 모델과 시스템을 도출하는 실용적 개발 방법론입니다.

## 배경

### 전통적 개발 방식의 문제

**시나리오 1: 끝없는 회의**
```
현업: "품질검사 프로세스 개선이 필요해요"
개발: "Entity 모델링부터 하죠"
     (화이트보드에 박스와 화살표)
개발: "이게 Aggregate고, 이게 Value Object고..."
현업: "???" (이해 못함)

결과: 2시간 후 아무것도 결정 안 됨
```

**시나리오 2: 요구사항 불일치**
```
현업: "검사 끝나면 자동으로 알림이 가야 해요"
개발: "네!" (메모함)

3개월 후 시연:
현업: "이게 아닌데... 승인까지 돼야 알림이 가는 거죠"
개발: "그런 얘기 없었는데요?"

결과: 다시 개발, 일정 지연
```

**시나리오 3: 추상화의 늪**
```
개발팀 회의:
"검사 결과가 Entity인가 Value Object인가?"
"Aggregate Root는 뭐지?"
"Bounded Context를 어떻게 나누지?"

결과: 현업은 이해 못하고, 개발팀만 논쟁
```

### Formology의 접근

**같은 상황, 다른 결과**
```
현업: "품질검사 프로세스 개선이 필요해요"
진행자: "어떤 문서들을 쓰고 계세요?"
현업: "검사의뢰서, 검사성적서, 불량기록지요"
진행자: "양식을 한번 그려볼까요?"

30분 후:
- 문서 목록 확정
- 양식 스케치 완료
- 업무 흐름 명확

결과: 즉시 개발 착수 가능
```

## 핵심 개념

### 1. 모든 것은 문서다

업무의 모든 커뮤니케이션은 문서로 표현 가능합니다:

| 현장 상황 | 문서화 |
|----------|--------|
| 통화 내용 | 통화기록지 |
| 구두 지시 | 구두지시확인서 |
| 회의 결과 | 회의록 |
| 카톡 메시지 | 업무연락지 |
| 이메일 | 업무연락서 |

**차이는 무엇인가?**
- 얼마나 공들여 쓰느냐
- 얼마나 오래 보관하느냐
- 얼마나 공식적이냐

### 2. 문서 분류 체계

```
문서 (Document)
 │
 ├─ 서(書): 공식 문서
 │  ├─ 의뢰서: "해주세요"
 │  ├─ 신청서: "허가해주세요"
 │  ├─ 지시서: "하세요"
 │  ├─ 보고서: "했습니다"
 │  └─ 확인서: "맞습니다"
 │
 ├─ 지(紙): 기록 용지
 │  ├─ 일지: 날마다 기록
 │  └─ 기록지: 사실 기록
 │
 ├─ 표(表): 목록/집계
 │  ├─ 현황표: 지금 상태
 │  └─ 집계표: 합산 결과
 │
 └─ 록(錄): 이력 모음
    ├─ 회의록: 회의 내용
    └─ 이력록: 변경 이력
```

### 3. 자연스러운 데이터 모델 도출

**핵심 통찰**: 양식(Form)의 구조를 보면 데이터 모델이 보인다!

```
양식 (Form):
┌─────────────────────┐
│ 품질검사의뢰서       │
├─────────────────────┤
│ [Main Section]      │
│ 제품명: [선택 ▼]   │  ← Field → FK: 제품 마스터
│ LOT번호: [입력]     │  ← Field → 속성
│ 수량: [숫자]        │  ← Field → 속성
├─────────────────────┤
│ [Child Section]     │
│ 검사항목:           │
│ ☑ 외관             │  ← Section → 1:N 테이블
│ □ 치수             │
└─────────────────────┘

자동 도출 (Record):
- Main Section → 테이블: qc_requests
- Field → FK: product_id
- Field → 속성: lot_number, quantity
- Child Section → 자식 테이블: qc_request_items
```

**개념 흐름**:
```
FormType (서식) → "품질검사의뢰서" 개념 정의
  ↓
Form (양식) → 구체적 레이아웃과 Section/Field
  ↓
Document (문서) → 유저가 작성한 실제 문서
  ↓
Record (레코드) → 데이터베이스에 저장
```

## Formology의 장점

### 현업에게

✅ **익숙한 용어**
- "검사의뢰서" (○) vs "InspectionRequest Entity" (×)
- 매일 쓰는 용어 그대로

✅ **구체적**
- 추상적 개념이 아닌 실제 양식
- 즉시 피드백 가능

✅ **빠른 검증**
- 문서 목록 보고 즉시 확인
- "이것도 필요해요!" 즉각 반영

### 개발자에게

✅ **명확한 스펙**
- FormType 목록 = 도메인 개념
- Form 구조 = API 스키마
- Document 흐름 = 워크플로우
- Section/Field = 데이터 구조

✅ **현업과 같은 언어**
- 소통 장벽 제거
- 오해 최소화
- Form/Section/Field 용어로 직관적 소통

✅ **자동 데이터 모델**
- Section → Entity 사상으로 ERD 자연 도출
- Field → Attribute 변환 규칙 명확
- 정규화가 자연스럽게 적용됨

## 방법론 비교

### vs 전통적 온톨로지/DDD

| 측면 | 온톨로지/DDD | Formology |
|------|-------------|---------|
| 시작점 | 추상적 개념 | 구체적 문서 |
| 용어 | Entity, Aggregate | 의뢰서, 보고서 |
| 이해도 | 개발자만 | 누구나 |
| 소요시간 | 수 주 | 수 일 |
| 현업 참여 | 인터뷰 대상 | 공동 설계자 |

### vs 전통적 데이터 모델링

| 측면 | 데이터 모델링 | Formology |
|------|--------------|---------|
| 시작점 | ERD | Form (양식) 구조 |
| 중간 단계 | Entity-Attribute | Section-Field |
| 사상 | 추상화 필요 | 직접 변환 |
| 정규화 | 명시적 (3NF, BCNF) | 암묵적 (자연 도출) |
| 검증 | 개발자 내부 | 현업 직접 (Form 기반) |
| 학습곡선 | 가파름 | 완만함 |

## 적용 범위

Formology는 다음과 같은 분야에 효과적입니다:

### ✅ 적합한 영역

- **제조업**: MES, CMMS, QMS
- **의료**: EMR, 진료 지원 시스템
- **물류**: WMS, TMS
- **사무**: 전자결재, 문서관리
- **연구**: LIMS, 연구노트 시스템

**공통점**: 문서/양식 기반 업무 프로세스

### ⚠️ 부분 적용

- **SNS/커뮤니티**: 일부 기능(글, 댓글)은 문서로 볼 수 있음
- **실시간 제어**: 문서보다는 이벤트 중심
- **게임**: 일부 기능(퀘스트, 아이템)만 적용 가능

### ❌ 부적합한 영역

- **순수 알고리즘**: 추천, 검색, AI
- **스트리밍**: 실시간 데이터 처리
- **IoT 센서**: 연속 데이터 수집

## 기대 효과

### 정량적 효과

실제 프로젝트 결과:

| 항목 | 전통 방식 | Formology | 개선율 |
|------|----------|---------|--------|
| 요구사항 정의 | 3주 | 2일 | 90% ↓ |
| 현업 이해도 | 40% | 95% | 137% ↑ |
| 요구사항 변경 | 주 3회 | 주 0.3회 | 90% ↓ |
| 데이터 모델 정확도 | 70% | 95% | 36% ↑ |

### 정성적 효과

**현업 반응**
> "우리가 하던 거 그대로 시스템화 됐어요!"
> "Entity가 뭔지 몰라도 시스템 개발에 참여할 수 있어요"

**개발팀 반응**
> "요구사항이 명확해서 개발이 빨라요"
> "현업과 소통이 원활해요"

**관리자 반응**
> "일정이 예측 가능해요"
> "품질이 올라가고 재작업이 줄었어요"
