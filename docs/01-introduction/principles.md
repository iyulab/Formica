# Formology 핵심 원칙

## 설계 철학

Formology는 다음 철학을 기반으로 합니다:

> **"추상화하지 말고, 구체화하라"**
> **"말하지 말고, 써라"**
> **"이론이 아닌, 현장을"**

## 10대 핵심 원칙

### 1. 구체성 우선 원칙

**원칙**: 추상적 개념보다 구체적 실체를 먼저

```
❌ 나쁜 예:
"Entity를 정의하고..."
"Aggregate를 설계하고..."

✅ 좋은 예:
"품질검사의뢰서 양식을 그려보고..."
"검사보고서에 어떤 내용이 들어가죠?"
```

**근거**: 현업은 구체적인 것을 이해하고, 구체적인 것에서 추상화는 자연스럽게 도출됨

**적용**:
- 워크숍에서 문서 이름부터 시작
- 양식을 먼저 그림
- 데이터 모델은 나중에 도출

---

### 2. 현장 언어 존중 원칙

**원칙**: 개발자 용어가 아닌 현장 용어 사용

```
❌ 나쁜 예:
"InspectionRequest Entity"
"QualityCheck Aggregate"
"DefectRecord Value Object"

✅ 좋은 예:
"품질검사의뢰서"
"검사성적서"
"불량기록지"
```

**근거**: 현업이 매일 쓰는 용어가 가장 정확하고 자연스러움

**적용**:
- 현업 용어 그대로 사용
- 번역하지 않음
- 코드 내부에서도 가능한 현장 용어 유지

---

### 3. 상향식 도출 원칙

**원칙**: 탑다운이 아닌 바텀업으로 설계

```
❌ 탑다운:
최상위 개념 정의
    ↓
중간 계층 설계
    ↓
구체적 구현
    ↓
현업: "이해 못함"

✅ 바텀업:
구체적 문서 나열
    ↓
패턴 발견
    ↓
자연스러운 분류
    ↓
현업: "맞아요!"
```

**근거**: 실무는 구체적 행위에서 시작되며, 추상화는 반복을 통해 발견됨

**적용**:
- 문서 목록부터 시작
- 패턴은 나중에 발견
- 분류는 자연스럽게 도출

---

### 4. 즉시 검증 원칙

**원칙**: 모든 산출물은 현업이 즉시 검증 가능해야 함

```
❌ 검증 어려움:
ERD 다이어그램
UML 클래스 다이어그램
온톨로지 그래프

✅ 즉시 검증:
"이 문서 쓰시죠?"
"양식이 맞나요?"
"흐름이 맞나요?"
```

**근거**: 빠른 피드백이 품질을 높이고 비용을 낮춤

**적용**:
- 워크숍 진행 중 즉시 확인
- "맞아요" / "아니에요" 즉각 반응
- 수정도 즉시

---

### 5. 자연스러운 진화 원칙

**원칙**: 완벽한 설계보다 점진적 개선

```
❌ 완벽 추구:
6개월 설계
    ↓
개발 시작
    ↓
"설계가 틀렸네" → 전체 재설계

✅ 점진적 개선:
기본 문서 도출 (2일)
    ↓
개발하며 발견
    ↓
"이 문서도 필요하네" → 자연스럽게 추가
```

**근거**: 업무는 살아있고, 완벽한 초기 설계는 불가능

**적용**:
- MVP 문서 세트로 시작
- 필요시 문서 추가
- 시스템이 자연스럽게 성장

---

### 6. 문서 자립성 원칙

**원칙**: 각 문서는 독립적으로 의미를 가져야 함

```
❌ 의존성 높음:
"이 데이터는 3개 테이블 조인해야 의미 파악"

✅ 자립성:
"품질검사보고서만 봐도 검사 결과 완전히 이해"
```

**근거**: 실제 종이 문서는 그 자체로 완결된 정보

**적용**:
- 문서 하나로 업무 완결
- 필요한 정보는 문서 안에 포함
- 참조는 보조 수단

---

### 7. 양식 우선 원칙

**원칙**: 데이터 모델보다 Form (양식) 구조 설계를 먼저

```
❌ 데이터 모델 먼저:
ERD 설계
    ↓
테이블 생성
    ↓
화면 설계
    ↓
"화면에 안 맞네" → 재설계

✅ Form (양식) 먼저:
FormType 정의 (개념)
    ↓
Form 구조 스케치 (Section/Field)
    ↓
현업 검증
    ↓
Section → Entity, Field → Attribute 사상
    ↓
"완벽!"
```

**근거**: Form의 Section-Field 구조에 모든 정보가 있고, Record 구조(ERD)는 사상 규칙으로 자동 도출

**적용**:
- FormType (개념) 정의부터
- Form (양식) Section/Field 스케치
- 입력 방식 정의 (Selection/Text/Numeric 등)
- ERD는 사상 규칙으로 도출

---

### 8. 실물 시뮬레이션 원칙

**원칙**: 종이 문서처럼 생각하고 설계

```
질문 목록:
□ 종이로 만든다면 어떤 양식?
□ 누가 작성하나?
□ 누가 승인하나?
□ 어디에 보관하나?
□ 얼마나 자주 쓰나?
□ 언제까지 보관하나?
```

**근거**: 종이 문서는 수백 년간 검증된 업무 도구

**적용**:
- 종이 양식 그리듯이
- 실제 업무 흐름 따라
- 물리적 제약 고려

---

### 9. 최소 기술 노출 원칙

**원칙**: 현업에게 기술 용어 노출 최소화

```
❌ 기술 노출:
"REST API 엔드포인트"
"정규화 3NF"
"Aggregate Root"

✅ 현업 언어:
"검사의뢰서 조회"
"중복 데이터 제거"
"주 문서"
```

**근거**: 현업은 기술을 몰라도 되고, 알 필요도 없음

**적용**:
- 워크숍에서 기술 용어 금지
- 설명은 현장 언어로
- 기술은 뒤에서 작동

---

### 10. 추적 가능성 원칙

**원칙**: 모든 것은 문서로 추적 가능해야 함

```
추적 가능해야 할 것:
✓ 누가 작성했나? → 작성자
✓ 언제 작성했나? → 작성일시
✓ 왜 작성했나? → 참조 문서
✓ 무엇을 했나? → 문서 내용
✓ 결과는? → 후속 문서
```

**근거**: 추적 불가능하면 관리 불가능

**적용**:
- 모든 문서에 메타정보
- 문서 간 연결 명확히
- 이력 관리

---

## 원칙 적용 우선순위

### P0 (필수)
1. 구체성 우선 원칙
2. 현장 언어 존중 원칙
3. 즉시 검증 원칙

**이유**: 이 3가지가 없으면 Formology가 아님

### P1 (강력 권장)
4. 상향식 도출 원칙
5. 양식 우선 원칙
6. 최소 기술 노출 원칙

**이유**: 실용성의 핵심

### P2 (권장)
7. 자연스러운 진화 원칙
8. 문서 자립성 원칙
9. 실물 시뮬레이션 원칙
10. 추적 가능성 원칙

**이유**: 품질과 유지보수성

## 원칙 충돌 해결

### 상황 1: 기술 효율 vs 현장 언어

```
문제: "품질검사의뢰서"는 너무 길어서
     변수명으로 부적절

해결:
- 외부(현업): "품질검사의뢰서"
- 내부(코드): "qc_request" 허용
- 단, 현업 화면에는 "품질검사의뢰서"
```

**원칙**: 현장 언어 > 기술 효율

### 상황 2: 자립성 vs 정규화

```
문제: 모든 정보를 문서에 포함하면 중복 발생

해결:
- 핵심 정보: 문서 안에 포함
- 참조 정보: 링크로 연결
- 붙임 정보: 스냅샷 복사
```

**원칙**: 자립성과 정규화의 균형

### 상황 3: 진화 vs 설계

```
문제: 무계획적 진화는 시스템 복잡도 증가

해결:
- 초기: 핵심 문서만 (MVP)
- 추가: 패턴 확인 후
- 정리: 주기적 리팩토링
```

**원칙**: 진화하되 방향성 유지

## 원칙 검증 체크리스트

### 워크숍 진행 중

```
□ 구체적인 문서 이름 사용하는가?
□ 현업이 이해하는 용어인가?
□ 추상적 개념을 강요하지 않는가?
□ 현업이 즉시 "맞아요" 할 수 있는가?
□ 기술 용어를 쓰지 않는가?
```

### 설계 검토 중

```
□ Form (양식) 구조에서 Section/Field가 명확한가?
□ Section → Entity 사상이 자연스러운가?
□ Field → Attribute 변환이 명확한가?
□ 각 Document가 독립적 의미를 가지는가?
□ 바텀업 접근을 따랐는가?
□ 현업 검증이 가능한가?
□ 추적 가능한 구조인가?
```

### 구현 검토 중

```
□ 화면 용어가 현장 언어인가?
□ API 이름이 FormType/Document 기반인가?
□ 데이터베이스가 Section→Entity, Field→Attribute 사상을 반영하는가?
□ Form 구조가 UI 레이아웃에 일관되게 반영되는가?
□ 진화 가능한 구조인가?
□ 기술이 현업에게 숨겨져 있는가?
```

## 반례: 원칙 위반 사례

### 반례 1: 추상화 먼저

```
❌ "검사 Entity를 먼저 정의하고..."

왜 틀렸나?
- 구체성 우선 원칙 위반
- 현업 이해 불가
- 즉시 검증 불가
```

### 반례 2: 기술 용어 노출

```
❌ 메뉴: "InspectionRequest CRUD"

왜 틀렸나?
- 현장 언어 존중 원칙 위반
- 최소 기술 노출 원칙 위반
- 현업 혼란
```

### 반례 3: 탑다운 강요

```
❌ "최상위 개념부터 정의합시다"

왜 틀렸나?
- 상향식 도출 원칙 위반
- 현업 참여 배제
- 검증 불가능
```
