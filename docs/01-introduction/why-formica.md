# 왜 문서중심인가?

## 눈손이 사고실험

### 제약을 통한 본질 발견

상상해보세요. 당신은 **눈손이**입니다.

```
    ●●
   ✋✋
```

**신체적 제약**
- ❌ 입이 없습니다 → 말할 수 없음 (회의 불가)
- ❌ 귀가 없습니다 → 듣지 못함 (설명 못 들음)
- ❌ 다리가 없습니다 → 움직일 수 없음 (현장 못 감)

**가능한 것**
- ✅ 큰 눈 → 문서를 읽을 수 있음
- ✅ 능숙한 손 → 문서를 쓸 수 있음

### 눈손이의 세계에서 일하기

이 상황에서 업무를 하려면 어떻게 해야 할까요?

**Q: 품질검사를 의뢰하려면?**
→ "품질검사의뢰서" 작성

**Q: 검사 결과를 알려면?**
→ "품질검사보고서" 읽기

**Q: 검사 기준은?**
→ "품질검사기준서" 참조

**Q: 불량을 발견하면?**
→ "불량기록지" 작성

**결과**
- 회의 없음!
- 구두 설명 없음!
- 오직 문서!

### 사고실험의 통찰

**핵심 깨달음**: 제약이 본질을 드러낸다

말과 회의에 의존하는 개발 방식은:
- 휘발성이 높음 (기억에 의존)
- 검증이 어려움 (말한/안한 논쟁)
- 추적이 불가능 (누가 언제 무엇을)

문서 중심 방식은:
- 영속성 (기록으로 남음)
- 검증 가능 (문서로 확인)
- 추적 가능 (누가/언제/무엇을 명확)

## 기존 방식의 근본적 문제

### 문제 1: 말(言)의 한계

#### 상황: 요구사항 회의

```
현업: "검사 끝나면 자동으로 알림이 가야 해요"
개발: "네, 알겠습니다" (메모장에 기록)

3개월 후 시연:
현업: "이게 아닌데... 승인까지 돼야 알림이 가는 거죠"
개발: "그런 말씀 안 하셨는데요?"
현업: "분명히 말했어요!"
```

**문제의 본질**
- 말은 휘발된다
- 기억은 왜곡된다
- 맥락이 소실된다
- 증거가 없다

#### 문서중심 접근

```
현업이 작성: "품질검사의뢰서 → 검사기록지 → 검사보고서 → 승인 → 알림"

개발이 확인: "이 흐름이 맞나요?"
현업: "네! 맞아요"

문서로 남음:
- 누가 (현업 김과장)
- 언제 (2024-11-01)
- 무엇을 (승인 후 알림)
```

**해결**
- 기록으로 남음
- 언제든 확인 가능
- 책임 소재 명확
- 추적 가능

### 문제 2: 추상화(抽象化)의 폭력

#### 상황: Entity 모델링 회의

```
개발: "검사 결과를 Entity로 모델링하겠습니다"
     (화이트보드에 박스와 선)
개발: "이게 Aggregate Root고..."
     "이게 Value Object고..."
     "Bounded Context는..."

현업: "???" (이해 못함)

2시간 후:
- 현업은 여전히 이해 못함
- 개발자끼리만 논쟁
- 아무것도 결정 안 됨
```

**문제의 본질**
- 현업과 개발자의 언어가 다름
- 추상화 수준이 현업 이해 범위를 벗어남
- 소통 단절

#### 문서중심 접근

```
진행자: "어떤 문서들을 쓰고 계세요?"
현업: "검사의뢰서, 검사성적서, 불량기록지요"
진행자: "양식을 그려볼까요?"

현업: (직접 그림)
┌─────────────────┐
│ 품질검사의뢰서   │
├─────────────────┤
│ 제품명:         │
│ LOT번호:        │
│ 검사항목:       │
└─────────────────┘

개발: "이해했습니다!"
```

**해결**
- 현업 언어 사용
- 구체적이고 명확
- 즉시 이해 가능
- 양방향 소통

### 문제 3: 탑다운(Top-Down)의 경직성

#### 상황: 시스템 설계

```
1주차: 도메인 모델 정의
2주차: Entity 관계 설계
3주차: 정규화 (3NF, BCNF)
4주차: 현업 검토

현업: "이거... 우리 업무랑 다른데요?"
개발: "하지만 이게 정석입니다"
현업: "정석은 모르겠고, 우리는 이렇게 안 해요"

결과: 전체 재설계
```

**문제의 본질**
- 이론이 실무를 압도
- 유연성 부족
- 현업 참여 배제

#### 문서중심 접근 (Bottom-Up)

```
1일차: 워크숍 - 문서 목록 도출
       "검사의뢰서, 검사성적서, 불량기록지..."

2일차: 양식 스케치
       현업이 직접 그림

3일차: 데이터 모델 도출
       양식에서 자연스럽게 ERD 생성

4일차: 현업 검증
       현업: "이거 맞아요!"

결과: 즉시 개발 착수
```

**해결**
- 실무가 이론을 이끔
- 자연스러운 진화
- 현업이 주도

## 문서중심의 본질적 우위

### 1. 구체성 (Concreteness)

**추상적 접근**
```
개발: "검사 프로세스의 상태 전이를..."
현업: "???"
```

**구체적 접근**
```
진행자: "검사의뢰서 쓰면 → 검사기록지 작성하고 →
        검사보고서 발행하죠?"
현업: "네, 맞아요!"
```

### 2. 검증 가능성 (Verifiability)

**말 기반**
```
현업: "그때 분명히 말했어요"
개발: "그런 말씀 없었는데요"
→ 증명 불가능
```

**문서 기반**
```
문서: "검사의뢰서 양식 v1.2 (2024-11-01)"
      "승인 후 알림 발송"
→ 명확한 증거
```

### 3. 진화 가능성 (Evolvability)

**탑다운**
```
초기 설계가 틀리면 → 전체 재설계
```

**바텀업**
```
문서 하나 추가 → 시스템에 자연스럽게 통합
"시정조치서가 필요해요" → "추가하겠습니다"
```

### 4. 참여 가능성 (Accessibility)

**Entity 모델링**
```
참여자: 개발자, 아키텍트
배제자: 현업 (이해 못함)
```

**문서 기반**
```
참여자: 현업, 개발자, 모두
배제자: 없음 (누구나 이해)
```

## 실제 효과 증명

### 사례 1: 플라스틱 사출 공장

**기존 시도 (Entity 모델링)**
- 소요: 2개월
- 결과: 현업 이해 못함, 프로젝트 중단

**Formica 적용**
- 소요: 2일 (워크숍) + 3개월 (개발)
- 결과:
  - 작업지시 누락 0건
  - 불량 추적 100%
  - 재고 정확도 99%
  - 현업 평가: "우리 하던 거 그대로!"

### 사례 2: 반도체 장비 CMMS

**기존 시도 (온톨로지 기반)**
- 소요: 4개월
- 결과: 온톨로지 복잡도 증가, 현업 참여 불가

**Formica 적용**
- 소요: 1일 (워크숍) + 6개월 (개발)
- 결과:
  - 장비 가동률 85% → 92%
  - 수리시간 4.5h → 3.2h
  - 부품재고 30% 감소

## 철학적 기반

### 오컴의 면도날 (Ockham's Razor)

> "필요 이상으로 많은 것을 가정해서는 안 된다"

**적용**
- Entity, Aggregate, Value Object 불필요
- 의뢰서, 보고서, 기록지로 충분

### 실용주의 (Pragmatism)

> "진리는 효용성으로 판단한다"

**적용**
- 이론적 완벽성보다 실무 적합성
- 추상화보다 구체화
- 탑다운보다 바텀업

### 현상학적 환원 (Phenomenological Reduction)

> "사물 자체로 돌아가라"

**적용**
- 현업이 실제로 하는 일로 돌아감
- 이론적 틀을 벗어남
- 본질에 집중

## 결론: 왜 문서인가?

### 문서는 가장 자연스러운 추상화

모든 업무는 결국 문서로 귀결됩니다:
- 구두 지시 → 지시서
- 회의 결과 → 회의록
- 검사 행위 → 검사보고서
- 승인 결정 → 승인서

**문서가 곧 업무의 본질입니다.**

### 문서는 현업과 개발자의 공통 언어

- 현업: 매일 쓰는 익숙한 용어
- 개발자: 명확한 스펙
- 관리자: 추적 가능한 증거

**모두가 이해하는 유일한 언어입니다.**

### 문서는 검증 가능한 명세

- Entity 모델: 검증 어려움
- 문서 목록: 현업이 즉시 확인 가능

**"이 문서 맞아요?"** → 1분 만에 검증 완료
