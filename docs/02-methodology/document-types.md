# 문서의 종류와 본질

## 모든 것이 문서다

### 포멀리티 스펙트럼 (Formality Spectrum)

업무의 모든 커뮤니케이션은 **포멀리티(formality)의 차이**일 뿐, 본질적으로 모두 문서입니다:

```
비공식 ←────────────────────────────────→ 공식
(informal)                           (formal)

구두지시   메모   카톡   이메일   의뢰서   계약서
   ↓       ↓     ↓      ↓       ↓       ↓
  암묵적  간단한  디지털  공식    승인    법적
  FormType FormType FormType FormType FormType FormType
    ↓       ↓     ↓      ↓       ↓       ↓
   양식    양식    양식   양식    양식    양식
    ↓       ↓     ↓      ↓       ↓       ↓
   문서    문서    문서   문서    문서    문서
```

**개념 체계 이해**:
- **FormType (개념양식/서식)**: 암묵적~법적, 각 포멀리티 수준에 따른 필수 요소 정의
- **Form (양식)**: FormType의 구체적 구현 (입력용, 보기용, 출력용 등)
- **Document (문서)**: 양식지를 유저가 작성한 것 (본 장의 주요 대상)

### 포멀리티의 세 가지 차원

모든 문서는 다음 세 가지 차원으로 구분됩니다:

| 차원 | 비공식 | 중간 | 공식 |
|------|--------|------|------|
| **작성 노력** | 간단히 | 표준 양식 | 엄격한 절차 |
| **보관 기간** | 임시 | 중기 | 장기/영구 |
| **공식성** | 구두 확인 | 업무 기록 | 법적 효력 |

### 일상의 모든 커뮤니케이션

| 커뮤니케이션 형태 | 양식 | 포멀리티 레벨 |
|------------------|------|--------------|
| 통화 내용 | 통화기록지 | 0-1 (암묵적) |
| 구두 지시 | 구두지시확인서 | 0-1 (암묵적) |
| 화이트보드 회의 | 회의 스케치 | 1 (간단함) |
| 카카오톡 메시지 | 업무연락지 | 1-2 (간단함~표준) |
| 이메일 | 업무연락서 | 2 (표준) |
| 공식 요청 | 의뢰서/신청서 | 2-3 (표준~높음) |
| 승인 문서 | 지시서/보고서 | 3-4 (높음~매우높음) |
| 법적 문서 | 계약서/증명서 | 5 (법적) |

**핵심 통찰**: 차이는 본질이 아니라 포멀리티 수준입니다.

## 지(紙)의 다양한 모습

### 본질: 모두 "지(紙)"의 보기 방식

```
Document (문서) = 기본 단위 (작성된 양식지)

표/록/철/편은 Document를 "보는 방식"일 뿐

Document (문서)
 │
 └─ 보는 방식에 따라:
     │
     ├─ 시간순 나열 → 표(表)
     ├─ 주제별 묶음 → 록(錄)
     ├─ 물리적 보관 → 철(綴)
     └─ 편집/선별 → 편(編)
```

**참고**: 실제로는 Record (DB 저장)가 이러한 집계/조회의 기반이 됩니다.

### 실전 예시: 같은 문서, 다른 보기

"작업일지"라는 Document가 있을 때:

| 보기 방식 | 명칭 | 설명 | 기술 구현 |
|----------|------|------|----------|
| 개별 | 지 | 2024-11-04 작업일지 | 단일 Document 조회 |
| 시간순 | 표 | 일일작업일지표 (11월 전체) | Records의 시간순 집계 |
| 제품별 | 록 | ABC제품 작업이력록 | Records의 주제별 필터링 |
| 연도별 | 철 | 2024년 작업일지철 | Records의 기간별 그룹핑 |
| 우수사례 | 편 | 모범작업 사례집 | Records의 조건별 선별 |

**핵심**: 같은 Record 데이터, 다른 View/Query!

## 문서 분류 체계

> **중요**: 서/지/표/록은 **Document (작성된 문서)** 의 분류입니다.
>
> - **FormType (서식)**: 포멀리티 수준에 따른 필수 요소 정의
> - **Form (양식)**: 구체적 레이아웃 (입력/보기/출력 최적화)
> - **Document (문서)**: 작성된 실제 문서 ← **서/지/표/록 적용 대상**
> - **Record (레코드)**: DB 저장 및 조회/집계 기반

### 서(書): 공식 문서

**포멀리티**: 레벨 2-5 (표준~법적)

**특징**
- 공식적 효력
- 승인 필요
- 법적/제도적 효력 가능
- 명확한 양식 준수
- FormType에 정의된 필수 Section/Field 포함

**종류**

| 접미어 | 의미 | 예시 | 포멀리티 | 특징 |
|--------|------|------|---------|------|
| 의뢰서 | "해주세요" | 품질검사의뢰서 | 2-3 | 요청 |
| 신청서 | "허가해주세요" | 휴가신청서 | 3 | 승인 필요 |
| 지시서 | "하세요" | 작업지시서 | 3-4 | 명령 |
| 보고서 | "했습니다" | 품질검사보고서 | 3-4 | 결과 전달 |
| 확인서 | "맞습니다" | 출하확인서 | 3-4 | 확인/증명 |
| 명세서 | "이렇습니다" | 거래명세서 | 3-4 | 상세 정보 |
| 계약서 | "약속합니다" | 품질보증계약서 | 5 | 법적 효력 |

### 지(紙): 기록 용지

**포멀리티**: 레벨 0-2 (암묵적~표준)

**특징**
- 사실 기록
- 승인 불필요 (대부분)
- 일상적 업무
- 간편한 작성
- FormType의 필수 요소 + 최소한의 부가 요소

**종류**

| 접미어 | 의미 | 예시 | 포멀리티 | 특징 |
|--------|------|------|---------|------|
| 메모 | 간단한 기록 | 업무메모 | 0-1 | 비공식 |
| 연락지 | 소통 내용 | 업무연락지 | 1-2 | 간단한 소통 |
| 일지 | 날마다 기록 | 작업일지 | 2 | 시간 기반 |
| 기록지 | 사실 기록 | 불량기록지 | 2 | 이벤트 기반 |
| 체크지 | 점검 항목 | 설비점검체크지 | 2 | 체크리스트 |

### 표(表): 목록/집계

**포멀리티**: 레벨 1-3 (간단함~높음)

**특징**
- 여러 Document의 집계 (View)
- Records의 시간순/항목별 정렬
- 빠른 조회 목적
- 집계/통계 기능
- 실시간 또는 정기 생성

**종류**

| 접미어 | 의미 | 예시 | 포멀리티 | 특징 |
|--------|------|------|---------|------|
| 현황표 | 지금 상태 | 재고현황표 | 2 | 스냅샷 |
| 집계표 | 합산 결과 | 일일생산집계표 | 2-3 | 통계 |
| 목록표 | 항목 나열 | 자재목록표 | 1-2 | 리스트 |

### 록(錄): 이력 모음

**포멀리티**: 레벨 2-4 (표준~매우높음)

**특징**
- 여러 Document의 이력 (Historical View)
- Records의 시간순 추적
- 검색 중요
- 장기 보관
- 추적 가능성 (Traceability)

**종류**

| 접미어 | 의미 | 예시 | 포멀리티 | 특징 |
|--------|------|------|---------|------|
| 회의록 | 회의 내용 | 품질회의록 | 2-3 | 회의 기록 |
| 이력록 | 변경 이력 | 설비이력록 | 3-4 | 변경 추적 |
| 대장 | 전체 관리 목록 | 거래처대장 | 3-4 | 마스터 |

## 중요도의 자연스러운 도출

### 빈도 × 중요도 매트릭스

```
         빈번함
           ↑
           │
    표     │     록
  (반복)   │   (모음)
           │
━━━━━━━━━━┼━━━━━━━━━→ 중요함
           │
    메모   │    계약서
  (일회성) │  (법적문서)
           │
```

### 워크숍에서 자연스럽게 나타남

**시나리오 1: 빈번함 발견**
```
현업: "매일 작업일지 쓰는데..."
→ 빈번함 → 표로 관리
→ 간소화 필요 (레벨 2, 표준 양식)
→ 모바일 입력 고려
```

**시나리오 2: 이력 중요성 발견**
```
현업: "불량 나면 계속 추적해야 하는데..."
→ 이력 중요 → 록으로 관리
→ 검색 기능 필수 (레벨 3-4)
→ 장기 보관 필요
```

**시나리오 3: 개별 문서 중요성 발견**
```
현업: "계약서는 한 번 쓰면 몇 년 보관하는데..."
→ 개별 중요 → 서로 관리
→ 승인 워크플로우 필요 (레벨 4-5)
→ 법적 효력 고려
```

**시나리오 4: 포멀리티 발견**
```
현업: "그냥 전화로 구두 지시하는데..."
→ 암묵적 양식 (레벨 0-1)
→ 필요 시 간단히 기록
→ 중요한 내용만 공식화
```

**결과**: 요구사항이 자연스럽게 도출됨!

## 문서의 포멀리티 레벨

### 레벨 분류

| 레벨 | 포멀리티 | 양식 | 예시 | 시스템 요구사항 |
|------|---------|------|------|----------------|
| **0** | **암묵적** | 불필요 | 전화, 구두지시 | 선택적 기록 |
| **1** | **낮음** | 간단 | 메모, 카톡 | 간단한 입력 |
| **2** | **중간** | 표준 | 의뢰서, 일지 | 양식 준수 |
| **3** | **높음** | 표준 | 지시서, 보고서 | 승인 필요 |
| **4** | **매우높음** | 엄격 | 검사보고서 | 다단계 승인 |
| **5** | **법적** | 법정 | 계약서, 증명서 | 법적 효력 |

### 레벨별 시스템 요구사항

**레벨 0-1: 비공식~낮은 포멀리티**
- ✓ 간단한 입력
- ✓ 최소 승인
- ✓ 수정 자유
- ✓ 짧은 보관
- ✓ 증거력 낮음

**레벨 2-3: 중간~높은 포멀리티**
- ✓ 양식 준수
- ✓ 승인 필요
- ✓ 수정 이력 관리
- ✓ 중기 보관
- ✓ 증거력 중간

**레벨 4-5: 매우높음~법적 포멀리티**
- ✓ 엄격한 양식
- ✓ 다단계 승인
- ✓ 수정 불가 (신규 발행)
- ✓ 장기/영구 보관
- ✓ 법적 증거 능력

### 워크숍 포멀리티 발견 질문

**질문 1: "얼마나 공들여 작성하나요?"**
```
현업: "그냥 카톡으로 간단히..."
→ 레벨 1-2, 간단한/표준 양식

현업: "엄격한 양식 따라 작성하고 검토받아요"
→ 레벨 3-4, 표준/엄격한 양식
```

**질문 2: "얼마나 오래 보관하나요?"**
```
현업: "며칠만 보고 버려요"
→ 레벨 0-1, 암묵적/간단한 양식

현업: "최소 5년, 법적으로 10년 보관해야 해요"
→ 레벨 4-5, 엄격한/법적 양식
```

**질문 3: "얼마나 공식적인가요?"**
```
현업: "팀 내부적으로만 쓰는 거예요"
→ 레벨 1-2, 낮은/중간 포멀리티

현업: "이게 법적 증거 자료가 돼요"
→ 레벨 5, 법적 포멀리티
```

### 포멀리티 진화 (Progressive Formalization)

같은 내용이 상황에 따라 포멀리티가 높아질 수 있습니다:

```
레벨 0: 전화 통화
   ↓ (문제 발생 시 공식화)
레벨 1: 통화기록 메모
   ↓ (중요도 상승)
레벨 2: 업무연락서 작성
   ↓ (승인 필요)
레벨 3: 공식 지시서
   ↓ (법적 분쟁)
레벨 5: 법정 증거 자료

예시: "제품 납기 변경"
1. 전화로 협의 (레벨 0)
2. 이메일 확인 (레벨 2)
3. 변경요청서 작성 (레벨 3)
4. 변경계약서 체결 (레벨 5)
```

## 참조 vs 붙임

> **구조적 이해**: 참조/붙임은 **Section 유형의 차이**입니다.
>
> - **Reference Section**: FK로 구현, 원본 Document/Record와 살아있는 링크
> - **Attachment Section**: 파일 또는 복제 데이터, 그 시점의 스냅샷

### 참조 (Reference) - 살아있는 링크

**특징**
- ✓ 항상 최신 내용
- ✓ 원본 수정되면 자동 반영
- ✓ 동적(Dynamic)
- ✓ DB: Foreign Key (FK)

**사용 시나리오**
- ✓ 현재 규정 참조 (레벨 2-3)
- ✓ 진행 중인 프로젝트
- ✓ 마스터 데이터
- ✓ Record 간 관계

**예시**
```
품질검사기준서 (Form/Document)
  └─ Reference Section:
      ├─ company_regulation_id (FK) → 회사 품질규정 Record (최신)
      └─ product_spec_id (FK) → 제품 사양서 Record (최신)
```

### 붙임 (Attachment) - 스냅샷

**특징**
- ✓ 그 시점의 복사본
- ✓ 원본 바뀌어도 영향 없음
- ✓ 정적(Static)
- ✓ DB: 파일 경로 또는 복제 테이블

**사용 시나리오**
- ✓ 승인 시점 자료 (증빙, 레벨 4-5)
- ✓ 계약 증빙 (레벨 5)
- ✓ 감사 대응 (레벨 4-5)
- ✓ 법적 증거 보존

**예시**
```
품질검사보고서 (Document, 2024-11-04, 레벨 4)
  └─ Attachment Section:
      ├─ 파일 테이블: 검사기준서_v1.3.pdf (복제)
      │               도면_rev2.pdf (복제)
      └─ 또는 복제 데이터: inspection_standard_snapshot (당시 버전 데이터)
```

### 참조 vs 붙임 선택 기준

| 기준 | 참조 | 붙임 |
|------|------|------|
| 변경 빈도 | 자주 변경 | 안정적 |
| 포멀리티 | 레벨 0-3 | 레벨 4-5 |
| 법적 효력 | 불필요 | 필요 |
| 증거 능력 | 불필요 | 필요 |
| 저장 공간 | 효율적 | 중복 허용 |
| 일관성 | 최신 유지 | 시점 고정 |

**실무 가이드**
```
참조 사용 (레벨 0-3):
- 제품 마스터 정보
- 현행 규정/절차
- 진행 중인 프로젝트

붙임 사용 (레벨 4-5):
- 승인받은 도면
- 계약 당시 견적서
- 감사 증빙 자료
```

**포멀리티와의 관계**: 높은 포멀리티(레벨 4-5)는 스냅샷(붙임) 필요, 낮은 포멀리티(레벨 0-3)는 참조 가능

## 실전 적용 패턴

### 패턴 1: 요청-응답

```
의뢰서 (요청, 레벨 2-3)
   ↓
기록지 (처리, 레벨 2)
   ↓
보고서 (결과, 레벨 3-4)

예시:
품질검사의뢰서 (레벨 3)
   ↓
품질검사기록지 (레벨 2)
   ↓
품질검사보고서 (레벨 4)
```

### 패턴 2: 지시-실행-보고

```
지시서 (명령, 레벨 3-4)
   ↓
일지 (실행, 레벨 2)
   ↓
보고서 (완료, 레벨 3)

예시:
작업지시서 (레벨 3)
   ↓
작업일지 (레벨 2)
   ↓
작업완료보고서 (레벨 3)
```

### 패턴 3: 신청-검토-승인

```
신청서 (요청, 레벨 3)
   ↓
검토의견서 (검토, 레벨 3)
   ↓
승인서 (결재, 레벨 4-5)

예시:
휴가신청서 (레벨 3)
   ↓
부서장검토의견서 (레벨 3)
   ↓
휴가승인서 (레벨 4)
```

### 패턴 4: 비공식-공식화 (Formalization)

```
전화/구두 (암묵적, 레벨 0)
   ↓ (필요 시)
메모/기록 (간단함, 레벨 1)
   ↓ (중요도 상승)
공식 문서 (표준~높음, 레벨 2-3)
   ↓ (법적 필요)
계약/증명 (법적, 레벨 5)

예시:
전화 협의 (레벨 0)
   ↓
업무연락지 (레벨 1-2)
   ↓
공식 의뢰서 (레벨 3)
   ↓
계약서 체결 (레벨 5)
```
