# 문서 이름 짓기

> **개념 맥락**: 이 문서는 FormType (서식 개념)의 이름을 짓는 방법을 다룹니다. FormType 이름이 Form (양식), Document (작성 문서), Record (레코드)의 이름 기반이 됩니다.

## 기본 공식

### FormType 네이밍 패턴

```
[대상] + [행위] + [접미어]

예시:
품질검사 + 의뢰 + 서 = "품질검사의뢰서" (FormType 이름)
  → Form: 품질검사의뢰서 양식 (레이아웃)
  → Document: 품질검사의뢰서 #QC-2024-001 (작성된 문서)
  → Record: qc_requests 테이블 (데이터베이스)

작업 + 지시 + 서 = "작업지시서" (FormType)
불량 + 기록 + 지 = "불량기록지" (FormType)
```

### 구성 요소

**대상 (Subject)**
- 업무 도메인
- 구체적이고 명확하게
- 예: 품질검사, 작업, 설비, 자재

**행위 (Action)**
- 동사형
- 업무의 목적
- 예: 의뢰, 지시, 기록, 확인

**접미어 (Suffix)**
- 문서 유형
- 서/지/표/록
- [문서 종류](document-types.md) 참조

## 접미어 체계

### -서(書): 공식 문서

**의뢰서**: "해주세요"
```
품질검사의뢰서
수리의뢰서
분석의뢰서
```

**신청서**: "허가해주세요"
```
휴가신청서
구매신청서
시정조치신청서
```

**지시서**: "하세요"
```
작업지시서
출하지시서
검사지시서
```

**보고서**: "했습니다"
```
품질검사보고서
작업완료보고서
사고보고서
```

**확인서**: "맞습니다"
```
출하확인서
검수확인서
재고확인서
```

**명세서**: "이렇습니다"
```
거래명세서
작업명세서
자재명세서 (BOM)
```

### -지(紙): 기록 용지

**일지**: 날마다 기록
```
작업일지
설비일지
안전순찰일지
```

**기록지**: 사실 기록
```
불량기록지
검사기록지
사고기록지
```

**체크지**: 점검 항목
```
설비점검체크지
안전점검체크지
청소점검체크지
```

### -표(表): 목록/집계

**현황표**: 지금 상태
```
재고현황표
생산현황표
인력현황표
```

**집계표**: 합산 결과
```
일일생산집계표
월별불량집계표
연간매출집계표
```

**목록표**: 항목 나열
```
자재목록표
거래처목록표
설비목록표
```

### -록(錄): 이력 모음

**회의록**: 회의 내용
```
품질회의록
안전회의록
경영회의록
```

**이력록**: 변경 이력
```
설비이력록
불량이력록
수정이력록
```

**대장**: 전체 관리 목록
```
거래처대장
자재대장
고정자산대장
```

## 네이밍 가이드라인

### 원칙 1: 현장 언어 우선

```
❌ 나쁜 예 (개발자 중심):
InspectionRequest
QualityCheckDocument
DefectRegistrationForm

✅ 좋은 예 (현장 중심):
품질검사의뢰서
품질검사서
불량기록지
```

### 원칙 2: 구체적으로

```
❌ 나쁜 예 (추상적):
관리서
처리지
확인표

✅ 좋은 예 (구체적):
재고관리대장
불량처리기록지
출하확인표
```

### 원칙 3: 일관성 유지

**같은 업무 도메인은 같은 용어**
```
✅ 일관적:
품질검사의뢰서
품질검사기록지
품질검사보고서

❌ 비일관적:
품질검사의뢰서
QC기록지
검사결과보고서
```

### 원칙 4: 간결성

```
❌ 너무 김:
제품품질최종검사결과보고서

✅ 적절:
품질검사보고서
또는
최종검사보고서
```

**단, 명확성 > 간결성**

## 메뉴/화면 이름

### FormType 기반 네이밍

```
❌ 개발자 중심 (CRUD 용어):
- 환자 추가
- 작업 등록
- 검사 입력

✅ 현장 중심 (FormType + 동사):
- 환자기록지 작성 (FormType: 환자기록지 → Document 생성)
- 작업지시서 발행 (FormType: 작업지시서 → Document 생성)
- 검사기록지 작성 (FormType: 검사기록지 → Document 생성)

**패턴**: [FormType 이름] + [업무 동사]
- 작성/발행/등록: Document 생성
- 조회/검색: Document 조회
- 수정/변경: Document 수정
- 승인/확인: Document 상태 변경
```

### 메뉴 구조 예시

**제조 MES**
```
생산관리
├─ 작업지시서 발행
├─ 작업일지 작성
└─ 생산실적표 조회

품질관리
├─ 검사의뢰서 작성
├─ 검사기록지 작성
└─ 불량기록지 조회

자재관리
├─ 자재요청서 작성
├─ 입고확인서 작성
└─ 재고현황표 조회
```

**병원 시스템**
```
진료
├─ 진료기록지 작성
├─ 처방전 발행
└─ 진료일지 조회

검사
├─ 검사의뢰서 작성
├─ 검사결과지 입력
└─ 검사이력록 조회
```

## 특수 케이스

### 케이스 1: 복합 문서

**패턴**: [대상1-대상2] + [행위] + [접미어]

```
자재-구매신청서
설비-수리의뢰서
제품-출하지시서
```

### 케이스 2: 세분화

**패턴**: [한정어] + [대상] + [행위] + [접미어]

```
긴급작업지시서
정기검사의뢰서
특별재고현황표
```

### 케이스 3: 버전/차수

**패턴**: Document 레벨에서 버전 관리

```
FormType: "품질검사보고서"
  → Document #1: 품질검사보고서 #QC-REP-2024-001 (v1.0)
  → Document #1: 품질검사보고서 #QC-REP-2024-001 (v2.0) ← 개정

FormType: "작업지시서"
  → Document: 작업지시서 #WO-2024-001 (1차)
  → Document: 작업지시서 #WO-2024-001 (2차) ← 재발행

**중요**: FormType 이름은 버전 없음, Document 인스턴스에만 버전 정보 부여
```

### 케이스 4: 임시/정식

```
임시작업지시서 → 작업지시서
가출하확인서 → 출하확인서
예비검사보고서 → 검사보고서
```

## 다국어 대응

### FormType → Record 영문 매핑 원칙

**한글 우선, 영문은 Record 레벨에서**
```
FormType: "품질검사의뢰서" (한글 - 현업용)
  ↓
Form: 품질검사의뢰서 양식 (한글 - 화면)
  ↓
Document: 품질검사의뢰서 #QC-2024-001 (한글 - 문서)
  ↓
Record: qc_requests 테이블 (영문 - DB)

구현 레이어별 네이밍:
- Frontend (화면): 품질검사의뢰서 (한글)
- API Endpoint: /qc-requests (영문 약어)
- Database: qc_requests (영문 snake_case)
- Code (변수): qcRequest (영문 camelCase)
- Code (클래스): QcRequest (영문 PascalCase)
```

### 영문 약어 가이드

| 한글 | 영문 전체 | 약어 |
|------|----------|------|
| 품질검사 | Quality Check | qc |
| 작업 | Work | work |
| 자재 | Material | mat |
| 설비 | Equipment | eq |
| 불량 | Defect | def |
| 의뢰 | Request | req |
| 보고 | Report | rpt |
| 지시 | Instruction | inst |

## 네이밍 체크리스트

### 워크숍 중 확인사항

```
□ 현장에서 실제로 쓰는 용어인가?
□ 개발자 용어를 억지로 번역하지 않았나?
□ 구체적이고 명확한가?
□ 같은 도메인 내에서 일관적인가?
□ 너무 길지 않은가? (7자 이내 권장)
□ 접미어가 문서 성격과 맞는가?
```

### 설계 검토 시 확인사항

```
FormType → Form → Document → Record 일관성:
□ FormType 이름이 명확하게 정의되었는가?
□ Form (화면) 제목이 FormType 이름과 일치하는가?
□ Document 인스턴스 표시가 FormType 기반인가?
□ Record (테이블명)이 FormType을 영문 변환한 것인가?

구현 레이어 일관성:
□ API 엔드포인트가 FormType 기반인가? (/qc-requests)
□ 변수/클래스명이 FormType 영문 변환과 일관적인가?
□ Frontend 화면 이름이 FormType 한글 그대로인가?
□ 다국어 대응 전략(한글→영문 매핑)이 명확한가?
```

## 네이밍 워크숍 프로세스

### Step 1: 브레인스토밍 (30분)

```
규칙:
- 현장에서 쓰는 용어 그대로
- 판단하지 말고 모두 적기
- 포스트잇에 하나씩

결과: 15~30개 문서 이름
```

### Step 2: 그룹핑 (20분)

```
비슷한 문서끼리 묶기:

[요청/의뢰]
- 품질검사의뢰서
- 자재요청서

[기록]
- 작업일지
- 불량기록지

[보고/확인]
- 검사보고서
- 출하확인서
```

### Step 3: 표준화 (15분)

```
접미어 통일:
"품질검사 요청" → "품질검사의뢰서"
"작업 일보" → "작업일지"
"불량 내역" → "불량기록지"
```

### Step 4: 검증 (10분)

```
현업에게 확인:
"이 이름 맞나요?"
"실제로 이렇게 부르시나요?"
"더 나은 표현 있나요?"
```
