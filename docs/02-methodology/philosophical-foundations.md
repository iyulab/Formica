# Formology 철학적 기반

## 개요

Formology는 단순한 방법론이 아닌 **인식론적 전환**을 제안합니다. 이 문서는 Formology가 기존 온톨로지 접근과 본질적으로 다른 이유를 철학적 관점에서 설명합니다.

> **핵심 주장**: 문서는 관점에 따라 달라지지 않는 **인식론적 앵커(Epistemic Anchor)**입니다.

---

## 온톨로지의 근본적 한계

### 1. 관점 의존성 (Perspectival Dependency)

**문제의 본질**: 같은 업무를 보는 관점이 다르면 모델도 달라짐

```
같은 "주문" 업무를 보고:

개발자 A의 모델:
Order
 └─ hasProduct
     └─ Product

개발자 B의 모델:
Customer
 └─ placeOrder
     └─ Order
         └─ containsItem
             └─ OrderItem
                 └─ refersToProduct
                     └─ Product

현업: "둘 다 이해 못하겠는데요?"
```

**철학적 분석**:
- 개념 모델링은 본질적으로 **주관적 해석** 과정
- 같은 도메인도 관점에 따라 다르게 모델링됨
- "정답"이 없어 끝없는 합의 과정 필요

### 2. 불가지론 함정 (Agnosticism Trap)

**현상**: 온톨로지 모델링에서 "올바른" 모델을 찾을 수 없음

```
Week 1: "Customer란 무엇인가?"
  → 구매한 사람? 잠재 구매자? 계약 당사자?
  → 합의 어려움

Week 2: "Order의 본질은?"
  → 거래? 의도? 계약? 이벤트?
  → 여전히 논쟁

Week 8: 여전히 기초 개념 정의 중...
```

**철학적 분석**:
- **불가지론**: "진리는 알 수 없다"
- 추상 개념의 "본질"에 대한 절대적 답이 없음
- 관점마다 다른 해석 → 합의 실패 → 프로젝트 지연

**Noy & McGuinness의 30초 규칙**:
> "도메인 전문가에게 30초 안에 개념을 설명할 수 없다면, 현재 요구사항에 비해 너무 복잡한 것입니다."

### 3. 커스터마이징의 끝없는 복잡성

**문제**: "우리 회사는 특별하다" → 온톨로지 커스터마이징 필요

```
일반 온톨로지
 ↓ customize
산업별 온톨로지
 ↓ customize
회사별 온톨로지
 ↓ customize
부서별 온톨로지
 ↓
...복잡도 폭발...
```

**철학적 분석**:
- 추상화 계층이 깊어질수록 현실과 괴리
- 메타 모델, DSL 도입 → 더 복잡해짐
- 실용성 감소, 유지보수 난이도 증가

## Formology의 철학적 우위

### 1. 문서의 객관성 (Objectivity of Documents)

**핵심 통찰**: 문서는 보는 시각에 따라 달라지지 않음

```
현실에 존재하는 물리적 실체:
📄 품질검사의뢰서

개발자 A가 보면: 품질검사의뢰서
개발자 B가 보면: 품질검사의뢰서
현업이 보면: 품질검사의뢰서
CEO가 보면: 품질검사의뢰서

→ 모두 동일하게 인지
→ 해석의 여지 없음
→ 관점 의존성 없음
```

**철학적 분석**:
- 문서는 **상호주관적(Intersubjective)** 실체
- 누구나 같은 것을 관찰 가능
- 주관적 해석이 아닌 객관적 관찰

### 2. 인식론적 앵커 (Epistemic Anchor)

**개념**: 문서는 인식의 확실한 출발점

```
온톨로지 접근:
"Order Entity가 존재한다"
→ 증명 불가능
→ 믿음의 영역
→ 합의 필요
→ 주관적

Formology 접근:
"주문서가 존재한다"
→ 지금 보여줄 수 있음
→ 경험의 영역
→ 합의 불필요
→ 객관적
```

**철학적 의의**:
- **물리적 실체 vs 추상적 개념**
- 검증 가능 vs 검증 불가능
- 경험적 증거 vs 이론적 구성

### 3. 간주관성 (Intersubjectivity)

**정의**: 서로 다른 주체들이 같은 대상을 공유하는 특성

```
온톨로지:
"내가 보는 Entity"와 "네가 보는 Entity"
→ 다를 수 있음
→ 주관적 개념들의 협상 필요

Formology:
"내가 보는 의뢰서"와 "네가 보는 의뢰서"
→ 같음
→ 공유된 현실, 협상 불필요
```

**실용적 효과**:
- 즉시 합의 가능
- 소통 비용 최소화
- 검증 가능한 기준

## 철학 전통과의 연결

### 플라톤 vs 아리스토텔레스

#### 온톨로지 = 플라톤적 이데아론

```
플라톤:
- 완벽한 "이데아"가 존재
- 현실은 불완전한 그림자
- 진리는 추상 세계에

온톨로지:
- 완벽한 "도메인 모델" 추구
- 현실 업무는 불완전
- 진리는 추상 모델에

문제:
→ 이상과 현실의 괴리
→ 끝없는 추상화
→ 현업과 단절
```

#### Formology = 아리스토텔레스적 실재론

```
아리스토텔레스:
- 실체는 현실에 존재
- 추상은 실체에서 유래
- 관찰 가능한 것이 진리

Formology:
- 문서는 현실에 존재
- 모델은 문서에서 도출
- 보이는 것이 진리

결과:
→ 현실과 모델 일치
→ 추상화 최소
→ 현업과 연결
```

### 합리론 vs 경험론

#### 온톨로지 = 데카르트적 합리론

```
합리론 (Rationalism):
"나는 생각한다, 고로 존재한다"

온톨로지:
"내가 개념화한다, 고로 모델이다"

과정:
생각 → 개념 → 추상화 → 모델

문제:
- 생각이 다르면 모델 다름
- 주관적
- 검증 어려움
```

#### Formology = 로크/흄의 경험론

```
경험론 (Empiricism):
"경험이 지식의 원천"

Formology:
"문서가 시스템의 원천"

과정:
관찰 → 문서 식별 → 양식 작성 → 모델 도출

장점:
- 누구나 같은 것 관찰
- 객관적
- 검증 가능
```

## 실전 비교: 품질검사 시스템

### 온톨로지 접근 (4주)

```
Week 1: 개념 정의
"Quality란 무엇인가?"
"Inspection의 본질은?"
"Defect의 존재론은?"

Week 2: 관계 정의
Quality hasProcess Inspection
Inspection detects Defect
Defect belongsTo Product

Week 3: 철학적 논쟁
개발자 A: "Defect는 Entity다"
개발자 B: "아니다, Value Object다"
개발자 C: "Event로 봐야 한다"

Week 4: 타협
"일단 Entity로..."
→ 현업: "그래서 언제 시작하나요?"
```

### Formology 접근 (1일)

```
오전 (2시간): 문서 나열
현업: "검사의뢰서, 검사기록지, 불량기록지"
→ 3개 문서 식별 완료

오후 (2시간): 양식 스케치
각 문서 어떻게 생겼나?
→ 양식 완성

다음날: 개발 시작
현업: "맞아요!"
개발: "바로 시작합니다"
→ 철학적 논쟁 없음
```

**시간 절감**: 95% (4주 → 1일)
**합의 성공률**: 온톨로지 <50% → Formology >90%

## 핵심 철학적 주장

### 주장 1: 문서는 자연스러운 추상화

**논거**:
- 모든 업무는 결국 문서로 귀결
- 구두 지시 → 지시서
- 회의 결과 → 회의록
- 검사 행위 → 검사보고서

**결론**: 문서가 곧 업무의 본질

### 주장 2: 문서는 공통 언어

**논거**:
- 현업: 매일 쓰는 익숙한 용어
- 개발자: 명확한 스펙
- 관리자: 추적 가능한 증거

**결론**: 모두가 이해하는 유일한 언어

### 주장 3: 검증 가능성이 진리성보다 중요

**실용주의 철학**:
- "진리는 효용성으로 판단한다"
- Entity 모델: 이론적으로 완벽해도 검증 어려움
- 문서 모델: 이론적 불완전해도 즉시 검증 가능

**결론**: 검증 가능한 것이 실용적 진리

## 방법론적 함의

### 1. 설계 철학의 전환

| 측면 | 온톨로지 | Formology |
|------|----------|---------|
| **출발점** | 추상 개념 | 구체적 문서 |
| **인식론** | 합리론 | 경험론 |
| **존재론** | 이데아론 | 실재론 |
| **진리 기준** | 이론적 완벽성 | 검증 가능성 |
| **합의 방식** | 철학적 논쟁 | 경험적 확인 |

### 2. 실천적 가이드라인

**원칙 1: 관찰 가능한 것부터**
```
❌ "주문 Entity의 본질은..."
✅ "주문서에 뭐가 들어가죠?"
```

**원칙 2: 검증 가능한 것만**
```
❌ "이 도메인 모델이 정확한가?"
✅ "이 문서 양식이 맞나요?"
```

**원칙 3: 공유된 현실에 기반**
```
❌ "내 개념 vs 네 개념"
✅ "우리가 보는 문서"
```

## 비판적 고찰

### Formology의 한계

**1. 복잡한 도메인 지식**
- 문서로 표현하기 어려운 암묵지 존재
- 추론 규칙이 필요한 경우
- → 해법: 제한적으로 온톨로지 보완 활용

**2. 시스템 간 통합**
- 표준화된 온톨로지 필요한 경우
- 다기관 지식 공유
- → 해법: 연동 레이어에서만 온톨로지 사용

**3. 학술적 엄밀성**
- 이론적 완벽성이 요구되는 연구
- → 해법: 연구 프로젝트는 온톨로지 사용

### 적용 범위

**Formology가 적합한 경우 (95%)**:
- ✅ 비즈니스 프로세스 디지털화
- ✅ 현업과의 빠른 소통 필요
- ✅ 실용적 ROI 요구
- ✅ 진화 가능한 시스템

**온톨로지가 필요한 경우 (5%)**:
- ✓ 국가 표준 준수 필수
- ✓ 다기관 지식 공유
- ✓ 장기 연구 프로젝트
- ✓ 복잡한 추론 필요

## 결론: 철학적 입장

### Formology의 철학적 정체성

**존재론**: 실재론 (Realism)
- 문서는 현실에 존재하는 물리적 실체

**인식론**: 경험론 (Empiricism)
- 문서는 관찰 가능한 경험의 대상

**방법론**: 실용주의 (Pragmatism)
- 검증 가능성이 진리성보다 중요

### 핵심 메시지

> **"온톨로지의 문제는 틀렸다는 것이 아니라, 대부분의 경우 과도하다는 것입니다."**

> **"문서는 누구나 같게 보는 유일한 실체입니다."**

> **"추상화하지 말고, 구체화하라. 말하지 말고, 써라."**
