# 양식 본질론

## 개요

이 문서는 **"양식이란 무엇인가?"**라는 질문에 답합니다. 철학적 깊이보다는 **실용적 이해**에 초점을 맞춥니다.

> **핵심 질문**: 품질검사의뢰서를 품질검사의뢰서로 만드는 것은 무엇인가?

---

## 1. 서식-양식-문서 구분

### 1.1 기본 2층 모델 (대부분의 경우 사용)

```
양식 (Form)          ← 설계 시점, 템플릿
  ↓ 인스턴스화
문서 (Document)      ← 실행 시점, 구체적 사례
```

**예시**:
```
양식: "품질검사의뢰서" 템플릿
문서: 2024년 11월 7일 김철수가 작성한 의뢰서 #QC-2024-001
```

**충분한 경우**:
- 양식 종류가 명확
- 변형이 적음
- 조직 규모 소~중형

### 1.2 확장 3층 모델 (필요 시만 사용)

```
서식 (Format)        ← 추상적 원형
  ↓ 구체화
양식 (Form)          ← 구체적 템플릿
  ↓ 인스턴스화
문서 (Document)      ← 실제 사례
```

**예시**:
```
서식: "검사의뢰서" (개념)
  ├─ 일반제품 검사의뢰서
  ├─ 식품 검사의뢰서 (유통기한 필드 추가)
  └─ 화학제품 검사의뢰서 (MSDS 필드 추가)
```

**필요한 경우**:
- 양식 변형이 많음
- 표준화 필요
- 대규모 조직

**원칙**: 복잡성은 필요할 때만 도입

## 2. 양식의 본질: 목적이 정체성을 결정

### 2.1 핵심 원칙

> **양식의 정체성 = 그것이 달성하는 업무 목적**

**예시**:
```
"주문서"와 "견적서"는 왜 다른가?

목적이 다르기 때문:
- 주문서: 구매 의사 확정
- 견적서: 가격 정보 제공
```

### 2.2 목적의 계층

**1차 목적: 직접적 업무 행위**
```
의뢰서 → 무언가를 요청
기록지 → 사실을 기록
보고서 → 결과를 전달
승인서 → 결정을 확정
```

**2차 목적: 조직적 기능**
```
- 증거 보존
- 책임 추적
- 지식 축적
- 프로세스 통제
```

## 3. 요소의 분류: 본질적 vs 부가적 vs 설명적

### 3.1 본질적 요소 (Essential Features)

**정의**: 없으면 목적 달성 불가능한 요소

**판별 질문**:
> "이 요소 없이 여전히 [양식명]인가?"
> → No이면 본질적

**예시**: 품질검사의뢰서
```
본질적:
- 제품명 (무엇을 검사?)
- LOT번호 (어떤 것을 검사?)
- 검사 항목 (무엇을 확인?)

→ 하나라도 없으면 검사 의뢰 불가능
```

**속성**:
- 필수 입력 (Required)
- 항상 표시
- 명확한 레이블

### 3.2 부가적 요소 (Accidental Features)

**정의**: 있으면 더 좋지만 필수는 아닌 요소

**판별 질문**:
> "이 요소가 있으면 더 편하거나 정확한가?"
> → Yes이면 부가적

**예시**: 품질검사의뢰서
```
부가적:
- 특이사항 (있으면 도움, 없어도 검사 가능)
- 참고사진 (추가 정보)
- 이전 검사 이력 (참고용)
```

**속성**:
- 선택 입력 (Optional)
- 접을 수 있음
- 보조 정보

### 3.3 설명적 요소 (Descriptive Features)

**정의**: 양식 사용을 돕는 메타 정보

**특징**:
- 입력하는 것이 아님
- 안내, 도움말, 예시

**예시**: 품질검사의뢰서
```
설명적:
- "LOT번호는 YY-MM-### 형식으로 입력하세요" (도움말)
- "제품명은 제품 마스터에서 선택" (안내)
- [?] 아이콘 (추가 설명)
```

**속성**:
- 읽기 전용
- 시각적으로 구분
- 필요 시 표시

## 4. 완비성과 적정성

### 4.1 완비성 (Completeness)

**정의**: 모든 본질적 요소가 포함됨

```
완비한 양식:
✓ 모든 본질적 요소 존재
✓ 목적 달성 가능
✓ 필요 최소 조건 충족

불완비한 양식:
✗ 본질적 요소 누락
✗ 목적 달성 불가
```

**체크리스트**:
- [ ] 모든 본질적 요소가 있는가?
- [ ] 각 요소가 명확하게 정의되었는가?
- [ ] 업무 목적을 달성할 수 있는가?

### 4.2 적정성 (Adequacy)

**정의**: 맥락에 맞게 사용 가능

```
적정한 양식:
✓ 완비함
✓ 과도하지 않음
✓ 사용하기 쉬움
✓ 효율적

부적정한 양식:
✗ 너무 복잡 (과도)
✗ 사용 어려움
✗ 비효율적
```

**체크리스트**:
- [ ] 현업이 쉽게 작성할 수 있는가?
- [ ] 입력 시간이 적정한가?
- [ ] 필드 수가 과도하지 않은가?
- [ ] 이해하기 쉬운가?

### 4.3 최소 충족성 원리

**원칙**: 필요한 최소한만 포함

```
이상적 양식 = 완비 + 최소

모든 본질적 요소 포함
불필요한 요소 제외
```

**실천**:
1. 본질적 요소: 반드시 포함
2. 부가적 요소: 비용-편익 분석 후 선택
3. 설명적 요소: 꼭 필요한 것만

**나쁜 예**:
```
✗ 사용하지 않는 필드 10개
✗ "혹시 나중에 필요할까봐" 추가
✗ 모든 가능한 정보 수집
```

**좋은 예**:
```
✓ 실제 사용하는 필드만
✓ 현재 목적에 필요한 것만
✓ 간결하고 명확
```

## 5. 실전 설계 가이드

### 5.1 양식 설계 4단계

**1단계: 목적 정의**
```
질문:
- 이 양식으로 무엇을 달성하는가?
- 누가 사용하는가?
- 언제 작성하는가?

예: "품질검사를 의뢰하기 위해"
```

**2단계: 본질적 요소 식별**
```
질문:
- 목적 달성에 반드시 필요한 정보는?
- 없으면 의미가 없는 필드는?

예: 제품명, LOT번호, 검사항목
```

**3단계: 부가적 요소 선택**
```
질문:
- 있으면 도움이 되는 정보는?
- 비용 대비 효과가 있는가?

예: 특이사항 (간단한 텍스트 필드)
```

**4단계: 설명 추가**
```
질문:
- 사용자가 어려워할 부분은?
- 어떤 안내가 필요한가?

예: LOT번호 형식 안내
```

### 5.2 검증 체크리스트

**완비성 검증**:
- [ ] 이 양식으로 목적을 달성할 수 있는가?
- [ ] 모든 필수 정보가 포함되었는가?
- [ ] 빠진 중요한 요소는 없는가?

**적정성 검증**:
- [ ] 현업이 5분 내에 작성 가능한가?
- [ ] 필드 수가 20개 이하인가?
- [ ] 불필요한 요소가 없는가?

**사용성 검증**:
- [ ] 현업이 이해할 수 있는 용어인가?
- [ ] 입력 방식이 직관적인가?
- [ ] 도움말이 충분한가?

## 6. 실전 예제

### 6.1 품질검사의뢰서 분석

**목적**:
```
1차: 품질검사 의뢰
2차: 검사 이력 추적
```

**본질적 요소**:
```
✓ 의뢰번호 (식별)
✓ 제품명 (대상)
✓ LOT번호 (특정)
✓ 수량 (범위)
✓ 검사항목 (내용)
✓ 의뢰자 (책임)
✓ 의뢰일시 (시점)
```

**부가적 요소**:
```
○ 긴급여부 (우선순위)
○ 특이사항 (참고)
○ 첨부파일 (증빙)
```

**설명적 요소**:
```
? LOT번호 형식: YY-MM-###
? 제품명: 제품 마스터에서 선택
? 검사항목: 표준 항목 또는 직접 입력
```

**검증**:
```
완비성: ✓ 검사 의뢰 가능
적정성: ✓ 3분 내 작성 가능
사용성: ✓ 현업이 이해 가능
```

### 6.2 나쁜 설계 vs 좋은 설계

**나쁜 설계**:
```
✗ 불필요한 필드 다수
   - 작성자 혈액형
   - 취미
   - 선호 색상 (???)

✗ 모호한 필드명
   - "항목1", "항목2", "항목3"

✗ 설명 없음
   - LOT번호 형식 모름
```

**좋은 설계**:
```
✓ 필요한 필드만
   - 제품명, LOT번호, 수량

✓ 명확한 필드명
   - "검사 대상 제품", "제조 LOT 번호"

✓ 충분한 안내
   - LOT번호 형식 예시
   - 제품명 선택 방법
```

## 7. 핵심 정리

### 7.1 핵심 원칙

1. **목적이 정체성**: 양식의 본질은 목적
2. **요소 분류**: 본질적/부가적/설명적 구분
3. **완비 + 최소**: 필요한 최소한만
4. **검증 가능**: 현업과 함께 확인

### 7.2 설계 체크리스트

**필수 체크**:
- [ ] 목적이 명확한가?
- [ ] 본질적 요소가 모두 있는가?
- [ ] 불필요한 요소는 제거했는가?
- [ ] 현업이 이해할 수 있는가?

**권장 체크**:
- [ ] 5분 내 작성 가능한가?
- [ ] 필드 수가 적정한가? (< 20개)
- [ ] 도움말이 충분한가?
- [ ] 테스트를 거쳤는가?

### 7.3 자주 하는 실수

**실수 1: 과도한 정보 수집**
```
✗ "나중에 필요할 것 같아서" → 대부분 사용 안 함
✓ "지금 필요한 것만" → 나중에 추가 가능
```

**실수 2: 모호한 정의**
```
✗ "기타", "비고", "참고" (무엇을 써야 하나?)
✓ "특이사항: 검사 시 주의할 점"
```

**실수 3: 현업 검증 없음**
```
✗ 개발자 혼자 설계 → 현업 사용 불가
✓ 현업과 함께 설계 → 즉시 사용 가능
```
