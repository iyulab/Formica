# 워크숍 진행 가이드

## 워크숍 개요

### 목표

2~3시간 워크숍으로:
- 15~30개 **FormType** (서식 개념) 목록 도출
- 주요 FormType의 **Form** (양식) 구조 스케치 (Section/Field)
- **Document** (문서) 흐름 정의
- **Section** 기반 데이터 관계 파악

**개념 체계**:
```
FormType (서식 개념) → 도출 대상 (품질검사의뢰서, 검사보고서 등)
  ↓
Form (양식 구조) → Section/Field 스케치 (Main Section, Child Section 등)
  ↓
Document (문서) → 업무 흐름 정의 (작성 → 승인 → 완료)
  ↓
Record (레코드) → 데이터 관계 파악 (Section → Entity 매핑)
```

### 준비물

```
✓ 화이트보드 또는 큰 종이 (최소 2개)
✓ 포스트잇 (3색 이상)
✓ 마커펜 (두꺼운 것)
✓ 카메라 (기록용)
✓ 2~3시간 확보
```

### 참석자

**필수**
- 현업 담당자 3~5명 (다양한 역할)
- 개발자 1~2명 (기록)

**최적 구성**
- 현업: 실무자 3명 + 관리자 1명
- 개발: BA 1명 + 개발자 1명
- 진행자: 1명 (Formology 이해자)

## 진행 단계

### Step 1: 오프닝 (10분)

**목표 공유**
```
진행자: "오늘은 어떤 문서들을 쓰시는지,
        그 문서 양식이 어떤지 정리하는 시간입니다."

"Entity, Domain Model 같은 어려운 얘기는 하지 않습니다."
"여러분이 실제로 쓰는 용어로 말씀해주세요."
```

**눈손이 소개 (5분)**
```
    ●●
   ✋✋

"상상해보세요. 여러분은 눈손이입니다."
"말할 수 없고, 들을 수 없습니다."
"오직 문서로만 일해야 합니다."

"품질검사를 의뢰하려면?"
→ "품질검사의뢰서"

"검사 결과를 알려면?"
→ "품질검사보고서"

"이렇게 모든 일을 문서로!"
```

**규칙 설명**
```
1. 기술 용어 금지 (Entity, Table, API 등)
2. 현장 용어 사용 (의뢰서, 보고서, 기록지)
3. 판단하지 말고 일단 모두 말하기
4. 틀린 답은 없음
```

### Step 2: 업무 선정 (10분)

**대상 업무 선택**
```
진행자: "어떤 업무를 분석할까요?"

좋은 예시:
✓ 품질검사 프로세스
✓ 구매 요청 프로세스
✓ 설비 고장 처리
✓ 불량 처리 흐름
✓ 생산 계획-실적 관리
```

**범위 정의**
```
시작점: "업무가 어디서 시작되나요?"
종료점: "어디서 끝나나요?"
경계: "다른 부서와의 연결점은?"
```

### Step 3: FormType 브레인스토밍 (30분)

**규칙**
```
1. 포스트잇에 FormType 이름 하나씩 (예: "품질검사의뢰서")
2. 마구 붙이기 (화이트보드에)
3. 판단하지 말고 일단 다 적기
4. 색깔로 구분 (선택)
   - 노랑: 요청/의뢰 (서)
   - 파랑: 기록 (지)
   - 초록: 보고/확인 (서)

**중요**: FormType 이름만 적기 (구조는 나중에)
```

**진행자 질문 기법**

**시작 질문**
```
"업무가 시작되려면 무슨 문서가 필요하죠?"
"누가 무엇을 요청하나요?"
```

**진행 질문**
```
"그 다음은?"
"실제 작업할 때 무엇을 쓰나요?"
"기록은 어떻게 하죠?"
```

**완료 질문**
```
"결과는 어떻게 알리나요?"
"확인은 누가 하나요?"
"승인은 필요한가요?"
```

**추적 질문**
```
"나중에 이력을 보려면?"
"이상 발생시 무엇을 쓰나요?"
```

**결과**
```
일반적으로 15~30개 문서 도출

예시 (품질검사):
- 품질검사의뢰서
- 검사기준서
- 검사기록지
- 검사성적서
- 합격확인서
- 불합격처리서
- 재검사의뢰서
- 불량기록지
- 시정조치서
- 예방조치서
- 검사일지
- 월별검사현황표
- 불량이력록
...
```

```
- 합격확인서

- 검사기준서
- 판정기준서
```

**진행 방법**
```
1. 포스트잇 이동하며 그룹 형성
2. 그룹 이름 붙이기
3. 중복 발견하면 통합
4. 빠진 것 발견하면 추가
```

### Step 5: 중요도 매트릭스 (15분)

**매트릭스 그리기**
```
        빈번함
          ↑
    표    │    록
  ┌───────┼───────┐
──┼───────┼───────┼──→ 중요함
  └───────┼───────┘
    메모  │   계약서
```

**배치 작업**
```
각 문서를 매트릭스에 배치:

우상단 (빈번 + 중요):
- 품질검사의뢰서
- 검사성적서
→ 록(이력)으로 관리

좌상단 (빈번 + 덜중요):
- 검사일지
- 체크리스트
→ 표(집계)로 관리

우하단 (중요 + 드묾):
- 검사기준서
- 품질계약서
→ 서(공식문서)로 관리
```

**자동 도출되는 요구사항**
```
빈번함 높음:
→ 모바일 입력
→ 간소화된 UI
→ 빠른 조회

중요도 높음:
→ 승인 워크플로우
→ 변경 이력 관리
→ 법적 증거 능력
```

**발견되는 것들**

```
- 승인 단계
- 예외 처리
```

### Step 7: Form 구조 스케치 (40분)

**중요 FormType 3~5개 선택**
```
기준:
✓ 가장 빈번한 것
✓ 가장 중요한 것
✓ 가장 복잡한 것
```

**Form (양식) 구조 그리기 - Section/Field 기반**
```
화이트보드에:

FormType: "품질검사의뢰서"
↓
Form: 품질검사의뢰서 양식

┌─────────────────────────┐
│ [Main Section] 의뢰 정보 │
├─────────────────────────┤
│ 의뢰번호: [자동]        │ ← Field: 자동생성
│ 작성자: [자동]          │ ← Field: 로그인 사용자
│ 제품명: [Selection ▼]  │ ← Field: FK to 제품 마스터
│ LOT번호: [Text]         │ ← Field: 텍스트 입력
│ 수량: [Numeric]         │ ← Field: 숫자 입력
├─────────────────────────┤
│ [Child Section] 검사항목 │ ← 1:N 관계
│ ☑ 외관                 │ ← Field: Checklist
│ □ 치수                 │ ← Field: Checklist
└─────────────────────────┘

→ Record 변환:
  Main Section → qc_requests 테이블
  Child Section → qc_request_items 테이블 (1:N)
```

**상세화 질문 (Section/Field 관점)**
```
Section 구조:
"이 정보들을 그룹으로 묶으면?" → Section 정의
"반복되는 항목이 있나요?" → Child Section (1:N)
"다른 문서를 참조하나요?" → Reference Section (FK)

Field 입력 방식:
"이건 선택하나요, 직접 입력하나요?"
→ Selection (FK) vs Text/Numeric

"선택이면 어디서 가져오나요?"
→ 제품 마스터, 고객 마스터 등 (FK 대상)

"필수인가요, 선택인가요?"
→ Field Required/Optional

Field 검증 규칙:
"숫자 범위는?" → Numeric Field 제약
"날짜 제약은?" → Date Field 제약
"중복 가능한가요?" → Unique 제약

Section 관계:
"이 Section이 다른 Section을 참조?" → Reference Section
"파일 첨부가 필요한가요?" → Attachment Section
```

---

### Step 8: Section 기반 데이터 관계 발견 (20분)

**Section 유형 확인**
```
1. Main Section vs Child Section
   "반복되는 정보 그룹이 있나요?"
   → "검사항목이 여러 개요" = Child Section (1:N)

2. Reference Section (FK)
   "어디서 선택하나요?"
   → "제품 마스터에서 선택해요" = Reference Section (FK)

3. Attachment Section (스냅샷/파일)
   "다른 문서를 첨부하나요?"
   → "검사기준서를 붙여요" = Attachment Section

4. Field 타입 확인
   "직접 입력하나요?"
   → Text/Numeric/Date Field
```

**자동 도출 (Section → Entity 사상)**
```
결과:
- FormType 목록 → Entity 후보
- Main Section → Main Entity (기본 테이블)
- Child Section → Child Entity (1:N 테이블)
- Reference Section Field → FK 컬럼
- Attachment Section → 복제 테이블 or 파일 저장

예시:
FormType: "품질검사의뢰서"
  → Form:
      [Main Section] → qc_requests 테이블
      [Child Section] 검사항목 → qc_request_items 테이블
      [Reference Field] 제품명 → product_id (FK)
```

### Step 9: 확인 및 정리 (15분)

**체크리스트**
```
□ 빠진 FormType 없나?
□ 중복 FormType 있나?
□ FormType 이름 명확한가?
□ Document 흐름이 맞나?
□ 현업이 모두 이해하는가?
□ Form 구조(Section/Field)가 실제와 부합하는가?
□ Section 기반 데이터 관계가 명확한가?
```

**우선순위 설정**
```
MVP (1차 개발):
- 필수 FormType 5~7개
- Form 구조 정의 (Section/Field)
- 핵심 Document 흐름만

2차:
- 추가 FormType
- 상세 Field 유효성 규칙
- 복잡한 Reference/Attachment Section

3차:
- 보고/통계용 FormType
- 고급 기능 (버전 관리, 승인 워크플로우)
```

## 진행자 가이드

### 해야 할 것

✅ **경청**
- 현업 말을 끊지 않기
- 모든 의견 존중

✅ **기록**
- 모든 내용 포스트잇/사진
- 누가 말했는지도 기록

✅ **중립**
- 특정 의견 편들지 않기
- 판단 유보

✅ **안내**
- 막히면 질문으로 유도
- 시간 관리

### 하지 말아야 할 것

❌ **기술 용어 사용**
- Entity, Table, API, ERD 금지
- FormType, Form (양식), Document (문서) 개념 사용 가능
- 현장 언어 우선 사용

❌ **완벽주의**
- "이것도 필요하지 않나요?" 금지
- 현업이 말한 것만

❌ **설계 강요**
- "이렇게 하는 게 좋습니다" 금지
- 현업이 결정

❌ **비판**
- "그건 이상한데요" 금지
- 일단 받아들이기

## 어려운 상황 대처

### 상황 1: 침묵

```
증상: 아무도 말을 안 함

대처:
1. 구체적 질문
   "아침에 출근하면 제일 먼저 뭐 하세요?"

2. 한 명씩 돌아가며
   "김과장님은 어떤 문서 쓰세요?"

3. 예시 제공
   "다른 공장에서는 작업지시서 쓰던데..."
```

### 상황 2: 의견 충돌

```
증상: A과장 vs B과장 다른 의견

대처:
1. 둘 다 기록
   "A과장님 의견도, B과장님 의견도 다 중요해요"

2. 차이 명확화
   "어떤 상황에서 A방식, 어떤 상황에서 B방식인가요?"

3. 선택 유보
   "개발하면서 둘 다 지원 가능한지 보죠"
```

### 상황 3: 너무 많은 FormType

```
증상: 50개 이상 FormType

대처:
1. 그룹핑 강화
   "이것들 합칠 수 있나요?"
   "같은 FormType의 다른 Form 아닌가요?"

2. 우선순위
   "당장 필요한 FormType만 추리면?"

3. 단계별 개발
   "1차는 핵심 FormType만, 2차에 나머지"
```

### 상황 4: 추상적 답변

```
증상: "프로세스 관리", "데이터 분석"

대처:
1. 구체화 질문
   "그걸 하려면 무슨 FormType (서식) 쓰세요?"
   "어떤 Document (문서) 작성하세요?"

2. 예시 요청
   "실제로 어제 뭐 쓰셨어요?"

3. Form (양식) 요청
   "그 문서 Form (양식) 보여주실 수 있어요?"
```

## 산출물

### 워크숍 직후

```
1. FormType 목록 (15~30개)
   - 포스트잇 사진
   - 그룹별 정리

2. FormType 그룹
   - 요청/의뢰 (의뢰서, 신청서 등)
   - 기록 (일지, 기록지 등)
   - 보고/확인 (보고서, 확인서 등)
   - 기준/규정 (명세서, 기준서 등)

3. Document 흐름도
   - 화이트보드 사진
   - 화살표 연결 관계 (Reference/Attachment Section)

4. 주요 Form (양식) 구조
   - 3~5개 Form 스케치
   - Section 구조 (Main/Child/Reference/Attachment)
   - Field 상세 정의 (Type, 입력 방식)

5. Section 기반 데이터 관계도
   - Main Section → Entity 목록
   - Child Section → 1:N 관계
   - Reference Section → FK 관계
   - Attachment Section → 복제/파일 관계
```

### 1주일 내 정리

```
1. FormType 목록 정제
   - Excel/Notion 정리
   - 우선순위 표시
   - 담당자 표시

2. Form 구조 상세화
   - Section 유형 명확화 (Main/Child/Reference/Attachment)
   - 각 Field 데이터 타입 (Text, Numeric, Selection, Date 등)
   - 필수/선택 구분
   - 검증 규칙

3. Section → Entity 사상 규칙 적용
   - Main Section → Main Entity (테이블)
   - Child Section → Child Entity (1:N 테이블)
   - Reference Section → FK
   - Attachment Section → 복제 테이블 or 파일 저장
   - 제약조건 정의

4. 개발 계획
   - MVP 범위 (핵심 FormType 5~7개)
   - 단계별 일정
   - 리소스 계획
```

## 성공 기준

### 워크숍 성공

```
✓ 현업이 주도적으로 참여
✓ 기술 용어 없이 진행 (Entity, ERD 대신 FormType, Form 개념 사용)
✓ 구체적 FormType 목록 도출
✓ 주요 Form 구조 스케치 완료 (Section/Field)
✓ 현업이 모두 이해
✓ 즉시 개발 착수 가능 (Section → Entity 사상 규칙 적용)
```

### 워크숍 실패

```
✗ 개발자가 주도
✗ Entity, Domain Model, ERD 논의로 시작
✗ 추상적 개념만 (FormType 없이 바로 데이터 모델 논의)
✗ 현업이 이해 못함
✗ Form 구조 없이 필드만 나열
✗ 추가 회의 필요
```
