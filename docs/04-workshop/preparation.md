# 워크숍 준비 가이드

## 개요

Formology 워크숍은 **2~3시간의 집중적인 협업 세션**으로, 현업과 개발팀이 함께 문서 목록을 도출하고 시스템 설계의 기초를 마련합니다.

### 워크숍 목표

```
Input:  업무 프로세스 (현업의 머릿속)
Output:
- 15~30개 문서 목록
- 주요 문서 양식 스케치 (3~5개)
- 문서 간 흐름도
- ERD 초안
- 우선순위 (MVP 범위)

시간: 2~3시간
```

---

## 사전 준비 (워크숍 1주 전)

### 1. 참석자 선정

#### 필수 참석자

**현업 (3~5명)**:
```
✓ 역할 다양성 확보
  - 관리자 1명 (의사결정권자)
  - 실무자 2~3명 (다양한 직무)
  - 현장 담당자 1명 (실제 사용자)

예시 (제조 MES):
- 생산 관리자 (의사결정)
- 품질 담당자 (실무)
- 현장 반장 (실무)
- 자재 담당자 (실무)
```

**개발팀 (1~2명)**:
```
✓ 진행자 1명
  - Formology 이해도 높음
  - 퍼실리테이션 경험
  - 현업 언어 존중

✓ 기록자 1명
  - 빠른 타이핑
  - 구조화 능력
  - 질문 능력
```

#### 피해야 할 구성

```
❌ 현업 1명만
   → 한 사람의 관점만 반영
   → 다양성 부족

❌ 개발자만 5명
   → 현업 의견 묻힘
   → 기술 용어 난무

❌ 경영진 참석
   → 현업이 솔직한 의견 못 냄
   → 정치적 발언 증가
```

---

### 2. 장소 및 준비물

#### 이상적인 장소

```
✓ 회의실 크기: 6~8인용
✓ 화이트보드: 최소 2개 (큰 것)
✓ 조명: 밝고 쾌적
✓ 소음: 조용함 (집중 필요)
✓ 환기: 2~3시간 집중 가능
✓ 인터넷: 필요 없음 (집중 방해)
```

#### 준비물 체크리스트

**필수**:
```
□ 화이트보드 2개 이상
□ 포스트잇 (5색, 각 100장)
  - 노란색: 요청/의뢰 문서
  - 파란색: 기록 문서
  - 초록색: 보고/확인 문서
  - 주황색: 표/현황 문서
  - 분홍색: 기타
□ 마커펜 (굵은 것, 5색)
□ 카메라/휴대폰 (기록용)
□ A4 용지 (양식 스케치용, 30장)
```

**선택**:
```
□ 간식 (쿠키, 과일)
□ 음료 (커피, 차, 물)
□ 타이머 (시간 관리용)
□ 노트북 (실시간 기록용)
```

---

### 3. 사전 공지 (1주 전)

#### 참석자에게 보내는 메일

```
제목: [Formology 워크숍] OO 시스템 문서 정의 세션 안내

안녕하세요,

OO 시스템 개발을 위한 Formology 워크숍을 아래와 같이 진행합니다.

■ 일시: 2024.11.15 (금) 14:00~17:00 (3시간)
■ 장소: 본사 3층 회의실 A
■ 참석자: 생산팀 3명, 품질팀 2명, 개발팀 2명
■ 목적: 업무 문서 목록 도출 및 양식 정의

■ 준비사항:
- 현재 사용 중인 문서 양식 (종이, Excel 등) 가져오기
- 업무 흐름에 대한 이해
- 특별한 IT 지식은 불필요합니다

■ 진행 방식:
1. 여러분이 실제로 사용하는 "문서"를 나열합니다
2. 문서의 양식을 간단히 스케치합니다
3. 문서 간 흐름을 정리합니다

※ Entity, Domain Model 등 어려운 용어는 사용하지 않습니다
※ 여러분의 업무 용어 그대로 사용합니다

궁금한 점은 언제든 문의해주세요.

감사합니다.
```

## 위험 요소 및 대응

### 위험 1: 시간 초과

**증상**:
- 브레인스토밍에서 1시간 소요
- 문서가 너무 많이 나옴 (50개+)

**대응**:
```
1. 타이머 사용 (각 단계별 제한)
2. "일단 다 적고, 나중에 우선순위 정하겠습니다"
3. 핵심 문서만 양식 스케치
4. 나머지는 다음 세션으로
```

---

### 위험 2: 기술 용어 난무

**증상**:
- 개발자: "이건 Aggregate Root인데..."
- 현업: "???"

**대응**:
```
1. 즉시 중단: "잠깐만요"
2. 현업 언어로 번역 요청
3. "현업 분들이 이해하실 수 있는 용어로 해주세요"
4. 규칙 재확인: "오늘은 현장 언어만 사용합니다"
```

---

### 위험 3: 의견 충돌

**증상**:
- A: "이건 작업지시서예요"
- B: "아니요, 생산계획서죠"

**대응**:
```
1. 둘 다 기록: "네, 일단 둘 다 적겠습니다"
2. 차이 명확화: "두 문서의 차이가 뭔가요?"
3. 필요시 둘 다 시스템에 반영
4. 판단 유보: "개발하면서 정하겠습니다"
```

---

### 위험 4: 침묵

**증상**:
- 아무도 말을 안 함
- 30초 이상 정적

**대응**:
```
1. 구체적 질문:
   "아침에 출근하면 제일 먼저 뭐 하세요?"
   "어제 어떤 문서 작성하셨어요?"

2. 한 명씩 지목:
   "김과장님, 평소에 어떤 양식 쓰세요?"

3. 예시 제공:
   "다른 공장에서는 작업지시서 쓰던데,
    여기도 비슷한 거 있나요?"
```

---

### 위험 5: 완벽주의

**증상**:
- "이 필드가 정확히 뭔지 정해야..."
- "모든 경우의 수를 고려해야..."

**대응**:
```
1. 진행 우선:
   "일단 대략적으로만 정하고,
    나중에 상세화하겠습니다"

2. MVP 강조:
   "완벽하지 않아도 괜찮습니다
    80%만 맞으면 됩니다"

3. 진화 가능성:
   "개발하면서 수정할 수 있습니다"
```

## 성공 기준

### 워크숍 성공 지표

```
✅ 정량적:
- 문서 목록 15~30개 도출
- 주요 문서 3~5개 양식 스케치
- 우선순위 합의 (MVP 5~7개)
- 시간 내 완료 (2~3시간)

✅ 정성적:
- 현업 100% 이해 ("우리 하던 거네요")
- 기술 용어 0% 사용
- 현업 주도적 참여
- 개발팀 명확한 이해

✅ 분위기:
- 긍정적 에너지
- 활발한 토론
- 웃음 있음
- "유익했다" 피드백
```

## 체크리스트

### 워크숍 1주 전
```
□ 참석자 선정 및 초대
□ 회의실 예약
□ 준비물 구매
□ 사전 공지 메일 발송
□ 참석 확인
```

### 워크숍 1일 전
```
□ 참석자 리마인더
□ 준비물 최종 점검
□ 회의실 재확인
□ 진행 시나리오 리허설
```

### 워크숍 당일 (30분 전)
```
□ 회의실 환기
□ 화이트보드 준비
□ 포스트잇, 마커 배치
□ 간식, 음료 준비
□ 카메라 준비
□ 휴대폰 진동 모드
```

### 워크숍 종료 직후
```
□ 화이트보드 사진 촬영
□ 포스트잇 정리 (버리지 말 것!)
□ 참석자 감사 메일
□ 산출물 디지털화 시작
```
