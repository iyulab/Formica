# 05-realization: 구현 원리 (Realization Principles)

## 디렉토리 목적

이 디렉토리는 Formology 방법론을 **구현**하기 위한 **원리와 이론**을 다룹니다.

**중요**: 이 디렉토리는 **특정 기술이나 코드를 제공하지 않습니다**. 대신, **기술 중립적인 구현 원리**를 제시합니다.

---

## 방법론 vs 구현 vs 실현

### 방법론 (Methodology)
- **위치**: `02-methodology/`
- **내용**: Formology의 핵심 개념과 방법
- **예시**: 서/지/표/록 분류, 양식-문서 구조, 참조-붙임-스냅샷

### 이론 (Theory)
- **위치**: `03-theory/`
- **내용**: 학문적 기반과 철학적 토대
- **예시**: 양식 본질론, 양식학 기반, 목적론

### 구현 원리 (Realization Principles) ← **이 디렉토리**
- **위치**: `05-realization/`
- **내용**: 방법론을 시스템으로 변환하는 원리
- **특징**: 기술 중립적, 보편적 적용 가능
- **예시**: 양식 → 엔티티 사상 원리, 인터페이스 설계 이론

## 핵심 원칙

### 원칙 1: 기술 중립성

이 디렉토리의 모든 문서는 **특정 기술에 종속되지 않습니다**.

```
❌ 잘못된 예: "React에서 useState를 사용하여..."
✅ 올바른 예: "상태 관리 원리는..."
```

### 원칙 2: 구조 보존

양식의 구조와 의미가 구현에서도 **보존**되어야 합니다.

```
양식 구조 ≅ 구현 구조

"동형 사상 (Isomorphism)"
```

### 원칙 3: 검증 가능성

모든 원리는 현업과 함께 **검증 가능**해야 합니다.

```
양식 (현업이 확인 가능)
  ↓ 사상 원리
구현 구조 (현업이 대응 확인 가능)
```

### 원칙 4: 점진적 구체화

추상적 원리에서 구체적 패턴으로 **점진적 세분화**가 가능합니다.

```
보편적 원리 (이 디렉토리)
  ↓
기술별 적용
  ↓
프로젝트별 구현
```

## 자주 묻는 질문

### Q1: 왜 코드가 없나요?

**A**: 이 디렉토리는 **원리**를 다룹니다. 원리는 기술 중립적이어야 하므로 특정 코드를 포함하지 않습니다.

### Q2: 실제로 어떻게 구현하나요?

**A**: 원리를 이해한 후, 프로젝트에 적합한 기술(SQL, MongoDB, React, Vue 등)을 선택하여 적용합니다.

### Q3: 이 원리는 모든 기술에 적용되나요?

**A**: 네. 관계형 DB, 문서 DB, 객체지향, 함수형 등 어떤 패러다임에도 적용 가능합니다. 표현만 다를 뿐, 원리는 동일합니다.

### Q4: 반드시 모든 원리를 따라야 하나요?

**A**: 아닙니다. 프로젝트의 제약과 요구사항에 따라 선택적으로 적용하되, **구조 보존**과 **검증 가능성** 원칙은 반드시 유지해야 합니다.
